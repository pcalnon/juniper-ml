- juniper-canopy

## Juniper Project

The juniper project is a research platform for investigating dynamic neural networks and novel learning algorithms.
The juniper project employs a microservices architecture that can be run as host-level services or as docker compose/kubernets managed containers.
The juniper project exposes a REST API with explicitely defined contracts and utilzes a websocket framework for near-realtime communications.
The juniper project is designed for high performance and is architectured for distributed processing using remote worker clients.
The juniper project implements concurent programming features using python's multiprocessing module with forkserver and advanced, interactive queueing.

## Background

The following juniper project applications have experienced significant regressions:

- Training is still stalling before accuracy / loss have reached their target values
  - This is a critical defect that blocks all further development
- There is currently a semantic error with the way juniper-canopy displays training progress:
  - an epoch should be defined as a single pass through the training data
  - an iteration, should be defined as one complete candidate node addition and one complete network output training.
    - as such, each iteration should be associated with an instance of network growth
- The data and boundary plots need to have their card heights increased to effectively make these plots larget without breaking their aspect ratios
- The process of updating network and training params is flakey
  - value updates are not always successful when submitted
  - the updated values are frequently not integrated into the status displays in the juniper canopy app

These regressions have caused the following, critical issues:

- Failing to complete training renders the entire project useless.  This is the single, most important requirement for the Juniper Project.
  - Failure to meet this requirement prevents any additional development work from being started
- semantic confussion with respect to epochs and iterations confuses the juniper-canopy display and makes the monitoring aspects of the app unusable
- problems with meta parameter updating via the juniper-canopy application destroys user trust in the application and leaves the network and network training in an ambiguous state.

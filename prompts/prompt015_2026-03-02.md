# documentation audit and update iteration

lets perform another, more rigorous and more comprehensive, documentation audit and update.

## criteria

the documentation audit should include the following:

**Validate links in documentation files:**

- broken internal document links
- broken links to other documentation files in the same application docs directory and sub dirs.
- broken links to external documentation links, to other applications
- broken external links

**Validate directory structures:**

- ensure application directory trees are correct
- ensure application sub-directory trees are correct
- ensure all paths listed in documentation files are correct and reflect the current directory layout
- ensure all referenced filenames are accurate and reflect the current naming scheme and location

**Validate Referenced Tools:**

- include application specific conda environments using pascal case names (e.g., juniper-cascor app should use JuniperCascor conda environment)
- include all installed toolsets, programs, and utilities documented in the application documentation files
- include installed mcp servers
- include local utility scripts

**Validate documented procedure:**

- Ensure all documented procedures reflect the actual state of the applications' configurations and codebases and use correct and up to date versions of:
  - documentation links
  - remote urls
  - referenced filenames
  - directory and subdir trees
  - local application paths
  - external application paths
  - toolsets, programs, and utilities
  - configurations
  - conda environments

**Validate Repo name replacements:**

this documentation audit and update iteration should also identify and address issues resulting from a project wide search and replace operation.
the search and replace resulting in replacing all instances of old repo names with the new repo names.

- old repo names:
  - JuniperCanopy
  - JuniperCascor
  - JuniperData
  - JuniperCascorWorker
  - JuniperCascorClient
  - JuniperDataClient
  - JuniperML
  - JuniperDeploy
- new repo names
  - juniper-canopy
  - juniper-cascor
  - juniper-data
  - juniper-cascor-worker
  - juniper-cascor-client
  - juniper-data-client
  - juniper-ml
  - juniper-deploy

while this is mostly correct, some instances of updated repo names should be reverted.
restore old repo names in contexts where documentation actually refers to the old repos (e.g., when discussing repo renaming)

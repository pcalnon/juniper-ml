# Juniper Project, Review and Document Cascor <-> Canopy Interface

Create a detailed plan to perform a rigorous and comprehensive code review and documentation of the full interface between the following Juniper project applications:

- juniper-canopy
- juniper-cascor

## Juniper Project

the juniper project is a research platform for investigating dynamic neural networks and novel learning algorithms.
the juniper project employs a microservices architecture that can be run as host-level services or as docker compose/kubernets managed containers.
the juniper project exposes a REST API with explicitely defined contracts and utilzes a websocket framework for near-realtime communications.
the juniper project is designed for high performance and is architectured for distributed processing using remote worker clients.
the juniper project implements concurent programming features using python's multiprocessing module with forkserver and advanced, interactive queueing.

## Background

### Requirements

These Juniper Project applications are being prepared for release and deployment.

Ongoing problems with the interface between these applications, both the way the interface has been implemented and how it is currently being used, have been the single greatest source of code quality issues and blockers for either of these applications and for the Juniper project itself.

To begin to address the root of these problems in a systematic way, a detailed code analysis and documentation of this interface should be performed.
The analysis and documentation are needed to provide a definitive description of the current state of the interface and code base.
The detailed documentation created, based on this analysis, should reflect both the interface's current state, and what the intended interface design actually is.
The interface description should include, but not be limited to, detailed descriptions of the following:

- data contracts
- code promises
- inteface agreements
- APIs
- developer and application expectations
- high-level code walk throughs
- detailed, operational level code walk-throughs

Documentation should also include explanatory diagrams illustrating the nature and the function of various components and aspects of the interface as well as the interface as a whole.
Included, explanatory diagrams should include, but not be limited to, the following:

- detailed field beakdowns including
  - field names
  - data types
  - default values
  - parent data structures and classes
- constants class locations of defaults

Documentation should clearly identify and describe any naming convention descrepancies, non-matching field names, or data types differences between different sides or different portions of the interface.
Code review and documentation should include a complete list of dataclasses, messages, data structures, fields and values included in the interface.

One area of interest is to identify discrepancies including:

- Any mapping classes, translators, or functions needed to transition data from one side of the interface to the other.
- Any specific requirements present on one side of the interface that are not present (or enforced) on the other

The codebase review and description should include a comprehensive list of code locations, in both applications, where the interface's data structures, messages, or classes are instantiated, initialized, are assigned values, or have values assigned.
Documentation shoudl also include a complete list of code locations where these interface components are used or referenced.

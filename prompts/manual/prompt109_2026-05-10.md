# live dataset switch

the desired behavior is more like a modified hybrid of options 1 & 3.

the swap_dataset_live() method should first pause the training being performed by the cascade_correlation.fit() method.
the new dataset should then be loaded from juniper-data via standard approach
the cascor model architecture should be modified as appropriate for the new dataset.

note that this should only, potentially, involve adding new nodes to the input & output layers if the new dataset geometry differs from the old dataset.
all existing hidden nodes should remain unchanged except for the addition of new (or removal of old) connections to and from the output layer and input layer nodes respectively.

it's also important to note that there are only two training modes for a cascor network:  candidate pool training and output training.

for output training mode, the training should resume with the possibly architecturally updated network continuing its output training with the initial minibatch (or epoch) of the new dataset.

for candidate pool training mode, the candidate nodes are trained against the error signal generated by the output layer.

the output layer error signal is generated by running an inference epoch and calculating the accuracy/error values after the current network's output training process has completed.
the resulting error signals for the dataset elements is then used to train the candidates in the pool.

when a new dataset is loaded during candidate training, however, that error signal becomes meaningless.
any further training against the old output layer error signal would be expected to cause significant accuracy degradation for all candidates in the pool and thus for the full cascor network itself when sub-optimal candidates are chosen for addition to the network.

this means that after the fit() operation has been paused, the new dataset loaded, and the network architecture updated, swap_dataset_live() should immediately transfer back to network output training mode using the new dataset elements.
the inner loop of the fit() method shouldn't be involved in this process.

the only open question is: how to transition back to output training mode
should the candidate training process be completed with selected node(s) added to the cascor network?
if not, then what if there are candidate nodes that are already within the correlation threshold values?
should the already usefully trained candidate node (or nodes) be added to the cascor network before control is returned?
or should all candidate pool nodes be abandoned and control transferred back to output layer training immediately?

what issues or concerns do you see in this approach and the reasoning behind it?
potential strengths, weaknesses, risks, and guardrails?

# Updated Design, Live Dataset Switch

## Purpose

The essential element that needs to be achieved, and the fundamental flaw in the previous design, is the preservation of the previously trained infrastructure.
In practice, this means that, after a live dataset switch has occurred and before any training on the new dataset takes place, the size-adjusted network should still be able to process elements of the old dataset with results that are essentially unchanged from their pre-altered-architecture state.

## Implementation

### What doesn't work

1. removing input or output layer nodes. this is a clear violation of the intention as it removes trained nodes that were necessary for the network to provide the previous dataset's results.
2. prepending a new input layer to the network with the number of input nodes determined by the new dataset record dimension. this approach also violates the need for the network to provide previous dataset's results prior to training on the new dataset.
3. naive addition of input layer nodes or output layer nodes to account for the new dataset's dimension or expected output dimension being greater than the original dataset or output dimension. this approach will render the network incapable of processing the old dataset records or producing the expected network output, respectively.

### Prerequisite Open Questions

While the new design is in process, there are some questions that need to be answered.

1. **Role of Input Layer nodes**:
During training, is an entire dataset record provided as input to each cascor network input layer node?
Or is the dataset record split up with a single dataset record element provided to each cascor network input layer node?
Please provide line number references to the applicable sections of the source code files.

2. **Backpropagation to Input Layer**:
During the initial output training phase, before any hidden nodes have been added to the network, is the output layer error backpropagated to the input layer to train the input layer weights?
Or do the randomized, input layer weights remain unchanged throughout the training process?

3. **Tensor Dimension Increase**:
Is there a compute performance friendly way to add a zero-initialized dimension to an existing torch tensor?

### Prerequisite Question Answers

1. **Q1 — Role of input layer nodes**:

There is no "input layer" in the traditional sense. The CascadeCorrelationNetwork has no input-layer neurons with their own weights — the dataset record is a passthrough. The entire record is provided to every downstream computational unit.

Evidence in src/cascade_correlation/cascade_correlation.py:

- Construction (line 714): self.output_weights = torch.randn(self.config.input_size, self.config.output_size, requires_grad=True) * self.random_value_scale — these are the output-layer weights with shape [input_size, output_size]. The rows that map
input → output live here, not in a separate input-layer parameter.
- _compute_hidden_outputs (line 1519, esp. lines 1539–1540): allocates buffer = torch.empty(batch_size, total_features), then buffer[:, :self.input_size] = x — the entire input record (all input_size features) is copied verbatim into the first
input_size columns of the working buffer. No transformation.
- Each hidden unit (lines 1541–1544): unit_input = buffer[:, :col] followed by torch.sum(unit_input * unit["weights"], dim=1) + unit["bias"]. Each hidden unit i reads all input_size + i prefix columns — i.e., all raw input features plus the outputs
of all earlier hidden units.
- The output layer (line 1577): output = torch.matmul(output_input, self.output_weights) + self.output_bias — operates on the same output_input buffer (input + all hidden outputs).

So the answer to your binary: the entire dataset record is provided to each downstream unit. "Split per input node" doesn't apply because there are no input nodes — the input record's columns are just the leftmost slots of the per-batch feature
buffer.

2. **Q2 — Backpropagation during initial output training**:

No "input layer weights" exist to be either trained or frozen. But the question maps cleanly onto the actual learnable parameters:

- During initial output training (no hidden units yet, so output_input == x), the only learnable parameters are output_weights (shape [input_size, output_size]) and output_bias (shape [output_size]).
- train_output_layer (line 1584) wraps these into a fresh nn.Linear (line 1630), copies the current values in (lines 1632–1634), runs the optimizer for epochs steps (lines 1660–1662 — loss.backward(); optimizer.step()), then copies the trained
weights back into self.output_weights and self.output_bias (lines 1674–1676).
- The input-to-output connection weights live in output_weights[0:input_size, :] and are absolutely updated by this training loop — backprop fully flows through them.

The randomized values from line 714 are the initial values of those input-to-output connections; they are not retained as frozen "input layer weights" — they're trained on the first call to train_output_layer.

What is frozen during output training: hidden unit weights. These are detached at promotion time (line 3568: weights.clone().detach()) and never touched again — the cascade-correlation contract.

3. **Q3 — Compute-friendly tensor dimension expansion**:

There is no torch primitive for true in-place dim expansion of an autograd-tracked parameter. Every safe option allocates a new tensor and memcpys the old data — O(new_size) time. They compile to equivalent CUDA/CPU kernels; the choice is
ergonomics, not throughput.

Four idiomatic options, all functionally equivalent:

```bash
┌──────────────────────────────────────────────────────────────────────────────────┬──────────────────────────────────────────────────────────────────────────────────┐
│                                      Idiom                                       │                                      Notes                                       │
├──────────────────────────────────────────────────────────────────────────────────┼──────────────────────────────────────────────────────────────────────────────────┤
│ torch.cat([t, torch.zeros(...)], dim=N)                                          │ Cleanest for pure-append (grow-output column-append). One C++ fused kernel call. │
├──────────────────────────────────────────────────────────────────────────────────┼──────────────────────────────────────────────────────────────────────────────────┤
│ torch.cat([t[:k], torch.zeros(...), t[k:]], dim=N)                               │ Cleanest for middle-insertion (grow-input row-insertion in P2-1c).               │
├──────────────────────────────────────────────────────────────────────────────────┼──────────────────────────────────────────────────────────────────────────────────┤
│ torch.nn.functional.pad(t, (l, r, top, bottom), mode="constant", value=0)        │ Sugar over the above. Pad-spec order is innermost-dim-first.                     │
├──────────────────────────────────────────────────────────────────────────────────┼──────────────────────────────────────────────────────────────────────────────────┤
│ new_w = torch.zeros(new_shape, dtype=t.dtype, device=t.device); new_w[:k, :] = t │ What P2-1c uses today. Explicit, two ops (alloc + copy).                         │
└──────────────────────────────────────────────────────────────────────────────────┴──────────────────────────────────────────────────────────────────────────────────┘
```

Performance: micro-benchmarks show torch.cat is 0–5 % faster on CUDA than the explicit zeros+slice for typical sizes (the kernel is fused), and indistinguishable on CPU for tensors under ~1 MB. For a once-per-swap operation this is unmeasurable.

**What does not work**:

- `tensor.resize_(...)` — in-place but does not zero-init new memory; contents are undefined. Use only if you immediately overwrite every new cell.
- Anything that tries to extend an autograd-tracked tensor mid-graph — `autograd` assumes parameter storage is stable per backward pass. Always do dim expansion between training steps (which P2-1c does, under `_training_lock`).

`requires_grad` is never preserved automatically — the new tensor must call .requires_grad_(True) explicitly. This is the only manual bookkeeping required regardless of idiom.

If P2-1d (or whatever you redesign it as) wants to refactor P2-1c's _grow_input / _grow_output to use torch.cat for cleaner code, the diff is purely cosmetic — no behavior or performance change.

The `_resize_network_output_for_datasets(..)` method is defined in the cascade_correlation.py source file at line ~731.

```python
    def _resize_network_output_for_dataset(self, input_size_new: int = 0, output_size_new: int = 0) -> None:
        """Resize the network output layer for a new dataset."""
        if output_size_new > self.config.output_size:
            padding_size_output = output_size_new - self.config.output_size
            self.output_weights = torch.cat([ self.output_weights, (torch.randn(self.config.input_size, padding_size_output) * self.random_value_scale)], dim=1)
            self.output_bias = torch.cat([ self.output_bias, (torch.randn(padding_size_output) * self.random_value_scale)], dim=0)
            self.config.output_size = output_size_new

        if input_size_new > self.config.input_size:
            padding_size_input = input_size_new - self.config.input_size
            self.output_weights = torch.cat([ self.output_weights, (torch.randn(padding_size_input, self.config.output_size) * self.random_value_scale)], dim=0)
            self.config.input_size = input_size_new
```

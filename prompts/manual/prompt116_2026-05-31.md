# juniper-recurse

There is a large development task that needs to be implemented that requires rigorous planning and design before it can be attempted.
This is a planning and design task only.
Changes to code should NOT be made during this planning process.

## Background

the juniper project is intended to be a research platform for investigating dynamic networks and novel learning algorithms.
to continue development in this direction, i'd like to develop a new application, juniper-recurse, that approximates the structure of juniper-cascor.
the new application should implement a recursive nn model.

## Design and Planning

there are three distinct components needed for this planning and design process.

### New Model Development

extensive investigation is necessary to evaluate various recursive neural network models.
the specific features that are desired include the following architectural characteristics:

- should be capable of handling time-series data
- should be capable of performing regressions, rather than just classifications, on datasets
- should be structurally amenable to dynamic operation
- should be fundamentally flexible wrt dataset types
- should prioritize problem-solving ability over simplicity
- bonus if it can be integrated with cascor networks

perform an in-depth analysis of available recursive models and the requirements list to determine the best 3 choices.
present the top three choices with a detailed breakdown of the strengths and weaknesses of each.
where applicable, include risks and guardrails associated with various elements of each potential model.

### Middleware Refactor

extensive analysis is necessary to identify all functionality, currently in juniper-cascor, that it would make sense to extract, to prevent having to add duplicate code to the new model.
all applicable, potentially common code should be extracted into a new or existing middleware application.
one or more new, middleware applications can be added to the juniper project, as necessary, to ensure that each application implements its intended function without any extraneous components.
for example:

- the juniper-canopy application should include all relevant aspects of the juniper front-end, monitoring and control gui.
- the juniper-cascor application should include all relevant aspects of the implementation of the Cascade Correlation neural network model.
- the juniper-data application should include all functionality required to obtain, generate, condition, reformat, load, modify, and save datasets used by any and all juniper project models.
- the juniper-observability (sub) application should handle all functionality needed to extract metrics and status from running juniper models and provide the corresponding data to the monitoring apps and resources.

the goal is to effectively create a template for the addition of new neural network models to the juniper project.
all functionality that wouldn't make sense to add for each new model should be moved to another application.

### Testing

the document should include, as a first class citizen, the testing architecture, components, toolsets, requirements, and functionality needed to ensure proper operation of:

- the new juniper-recurse application
- any new middleware application(s) created
- any modifications to existing applications

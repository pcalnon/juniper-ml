# juniper-recurrence

The next juniper project model to be implemented is the juniper-recurrence model.

## Background

The literature review, architecture evaluation, and POC verification that were involved in evaluating various model structures and architectures have wrapped up.

The initial design direction had a strong bias towards implementing recurrence.
This represented an apparent limitation of a large set of model designs considered, the majority of which were functionally limited to modeling only star-free formal languages.

A key discovery made during the investigation, however, was that **the star-free ceiling is a property of the dynamical-state substrate, not of capacity, depth, output richness, or memory length.**
This was particularly relevant after evaluating all current and planned juniper project datasets.

### Context

The highest priority work at this point, is a research, synthesis, and design task.
This task will focus on the selected model, and perform a deep and more detailed analysis of the approach.
Strengths, weaknesses, risks, and guardrails should be noted and analyzed in detail.
Implementation challenges and the complex, challenging, or larger scale integrations with the other juniper project applications should be documented in detail as well.

### Conclusion

#### Model Type

The model design that was found to best handle all currently implemented and planned datasets for the Juniper Project is referred to in the documentation is the following:

**P3-C (LMU + Approach-C)**: A closed-form variable-Δt LMU discretization with the following properties:

- PROs:
  - is C1-CLEAN:
    - matrix-exp of a fixed Legendre matrix — A
    - B not learned
    - not data-dependent
  - no large, mathematically complex, and poorly recipe'd structures required
    - no ODE solver
    - no auto-diff-through-solver
  - provides the strongest bias for the Δt signal
  - is the most transparent recurrent option
    - is transparent without being trivial
  - has the longest controllable horizon (window θ)
  - includes continuous-time LMU
  - its Δt-handling is the best of all options considered
    - the dt channel being the discretization step.
  - delivers strongest irregular-Δt capability
    - grid-invariance:
      - e_reg=0.035,
      - e_irr=0.039-0.043,
      - ~1.15×
  - is genuinely recurrent

- CONS:
  - it does inherit the star-free ceiling, however:
    - this is not an issue for all current and planned datasets
      - e.g., the equities dataset, which implements forecasting, is not a counting task
  - poc's used during model evaluation had no fixed-Δt negative control
    - so "fixed-Δt fails the e_irr bound" is an analytic assertion not a measurement
  - implementation as a grown structure is a critical and as-yet-still-open question

#### Continuous time handing

**Δt specific approaches:**

A. Δt-as-input
    - cleanest
    - models unchanged
    - weak bias signal
B. Δt-gated exp decay
    - strong real-time-forgetting bias
    - perturbs ESP/freeze
C. solver-free continuous-time LMU
    - strongest bias
    - LMU-only
    - empirically verified
D. resample + impute
    - lossy
    - data-side
    - loses informative-sampling signal

### Blockers

As a simpler first step, this specific node type can be instantiated initially as a standalone fixed-order model behind the common interface
As a next step, it can be implemented as a grown, cascade unit.

Deploying it fixed-order sidesteps the two unresolved blockers:

- the homogeneous-pool growability gap
- the worker ndim>2 cap

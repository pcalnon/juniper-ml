# juniper-canopy issues: prompt creation

## background

prior work has resolved the issues that prevented cascor model training from being started in the canopy front-end.
however, the juniper-canopy application is still experiencing a number of issues--previously shadowed by the training start problem--during various stages of the cascade correlation training process.

## description

### overview

- training is launching successfully in canopy and being completed successfully
- the canopy web page is not being refreshed to reflect the ongoing training progress
- numerous fields, metrics, and status indicators remain stale throughout the training process
- the cascor network graph displayed in the "Network Topology" tab, is not being updated in real time and becomes stale as training progresses
- the "Decision Boundary" plot, however, DOES appear to be updated in real-time throughout the training process
- a manual page refresh is required to update juniper canopy's display of stale training statuses and parameter values
- parameter updates through canopy appear to be working, but the "Failed to apply (502)" error message is still being displayed with the "Apply Parameters" button is clicked
- switching to a new dataset appears to be working for most 2-D datasets, e.g., spirals, xor, moons, circles.
- however, training fails to start when the MNIST dataset is selected for training with the following error message displayed:
  - "Start failed. HTTP 409: Training could not be started: Training cannot be started: juniper-data fetch failed: Request failed (500): Internal server error."

## deliverables

generate a prompt for development of a planning document addressing the identified issues

### prompt constraints

- the prompt should be generated by custom agents as appropriate
- use custom agents to generate, validate, and test the prompt
- the prompt should be detailed, correct, and effective
- the prompt should drive the creation of a planning document
- the prompt should be written to a markdown file in the prompts/generated/ directory
- the prompt should be named in accordance with the current naming convention:
  - JUNIPER_[Application Name: e.g., CANOPY, CASCOR, DATA, etc.]_[Prompt Description]_[Purpose: e.g., PLAN, IMPLEMENT, TASK, DEBUG, etc.]_[YYYY-MM-DD_hhmm].md
- the prompt should be syntactically correct and linted successfully

### planning document constraints

- the planning document should be generated using custom agents as appropriate
- the planning document should address, at a minimum, all currently identified issues with juniper-canopy
- all aspects of the planning document should be validated using multiple, independent sub-agents with a specific focus on preventing hallucinations
- the planning document should be written to a markdown file in the notes/ directory
- the planning document should be syntactically correct and linted successfully

## fable-5 model guardrails

- this task involves prompt generation, and the prompt itself is only driving planning document creation.
- there should NOT be any aspect of this work that develops dual-use, proof-of-concept exploit code.
- as such, all work for this task should be carried out using fable-5.
- if there are any specific task components that fable-5 is unable to perform, only that specific task component should be handed off to opus-4.8.
- task components that must be handed off to opus-4.8 should be should be broken down to the most granular level possible to precisely include only the excluded work.

## Remaining Work (in priority order)

### 1. juniper-ml — Actions allowlist policy (URGENT, blocks PyPI)

**Root cause** (captured 2026-04-10 via WebFetch on <https://github.com/pcalnon/juniper-ml/actions/runs/24228468143>):

> "The actions `actions/checkout@de0fac2e...`, `actions/setup-python@a309ff8b...`, `actions/upload-artifact@bbbca2dd...`, `actions/download-artifact@3e5f45b2...`, and `pypa/gh-action-pypi-publish@ed0c5393...` are not allowed in `pcalnon/juniper-ml` because all actions must be from a repository owned by `pcalnon`."

**Fix** (one-time, GitHub UI only — NOT a code change):

1. Open <https://github.com/pcalnon/juniper-ml/settings/actions>
2. Under **Actions permissions**, choose either:
   - **"Allow all actions and reusable workflows"** (simplest), OR
   - **"Allow `pcalnon` actions and reusable workflows, and select non-`pcalnon` actions and reusable workflows"** with allowlist:

     ```bash
     actions/*
     pypa/gh-action-pypi-publish@*
     ```

3. Save
4. Re-run failed runs OR push a no-op commit. Specifically, re-run `https://github.com/pcalnon/juniper-ml/actions/runs/24228468143` (Publish to PyPI for v0.4.0).
5. Verify `juniper-ml==0.4.0` lands on PyPI (`pip index versions juniper-ml` or check <https://pypi.org/project/juniper-ml/>).

**Impact**: Without this fix, `juniper-ml v0.4.0` exists as a GitHub Release but is NOT installable via `pip install juniper-ml==0.4.0`. Same restriction also breaks `CI/CD Pipeline` on every PR.

**Important**: This is a settings change, not a code change. **Do not** create a PR for this. If you also want to apply the same allowlist to `juniper-cascor-client` and `juniper-cascor-worker` (verify whether they have the same restriction by inspecting recent run failures), do those at the same time.

### 2. juniper-cascor-client + juniper-cascor-worker — Security Scans job

**Two distinct sub-issues in the same job**, both visible in CI logs from runs `24216667630` (cascor-client) and `24216671263` (cascor-worker):

**(a) Bandit format flag**: Workflow invokes bandit with `--format sarif`, but the installed bandit only supports `csv/custom/html/json/screen/txt/xml/yaml`. Error: `bandit: error: argument -f/--format: invalid choice: 'sarif'`. Subsequent step fails with `Path does not exist: reports/security/bandit.sarif`.

**Fix options**:

- Pin bandit to a version that supports SARIF (≥1.7.x with `bandit-sarif-formatter` extra), OR
- Switch to `--format json` and post-process to SARIF, OR
- Drop the SARIF output requirement

**(b) pip-audit on unpublished local package**: pip-audit tries to audit `juniper-cascor-client (0.3.0)` (and `juniper-cascor-worker (0.3.0)`) and fails because neither is on PyPI yet. Error: `Dependency not found on PyPI and could not be audited: juniper-cascor-client (0.3.0)`.

**Fix options**:

- Add `--ignore-vuln` or skip-self via `pip-audit --skip <name>` (check current pip-audit docs)
- Use a `requirements.txt`/`requirements.lock` based audit instead of project-based, excluding self
- Pin pip-audit version that handles unfound packages gracefully

Both sub-issues exist on `main` (verified — recent main runs all `failure`). Fix in one PR per repo.

### 3. juniper-cascor-client — Documentation Links job

**Root cause**: 3 broken cross-repo links in pre-existing files. CI runs `check_doc_links` without `--cross-repo skip`, so it tries to follow `../../juniper-ml/...` paths that don't exist in the CI checkout:

```bash
docs/DEVELOPER_CHEATSHEET.md:187: broken link [Ecosystem Cheatsheet](../../juniper-ml/notes/DEVELOPER_CHEATSHEET.md) -> file not found
docs/DEVELOPER_CHEATSHEET.md:188: broken link [juniper-data-client Cheatsheet](../../juniper-data-client/docs/DEVELOPER_CHEATSHEET.md) -> file not found
docs/DOCUMENTATION_OVERVIEW.md:104: broken link [Juniper Ecosystem Guide](../../CLAUDE.md) -> file not found
```

**Fix**: Update the `Documentation Links` step in `.github/workflows/ci.yml` to invoke `check_doc_links` with `--cross-repo skip` (matches the convention used by juniper-ml, juniper-data, and juniper-data-client). Verify after edit that the local non-cross-repo links still validate cleanly.

## Root Cause Analysis

### Bug 1: `generate_uuid()` does not output UUID to stdout (PRIMARY — affects 10/10 failures)

**File**: `scripts/wake_the_claude.bash`, lines 160-184

**Problem**: The `generate_uuid()` function generates a UUID into a local variable, validates it
with `is_valid_uuid()`, logs a debug message, and returns `TRUE`/`FALSE`. However, it never
`echo`s the UUID to stdout. The caller on line 476 captures stdout:

```bash
generated_uuid="$(generate_uuid)"
```

Since `generate_uuid()` never echoes the UUID, the captured value contains only `debug_log`
output (e.g., `"wake_the_claude: Generated UUID: <uuid>"`). This polluted string is then:
1. Stored as the session ID value
2. Passed to `save_session_id()` which tries to extract a UUID via `awk`
3. `awk` extracts `"wake_the_claude:"` as the second field — not a valid UUID
4. `is_valid_uuid` fails → script exits with usage error

**Evidence**: Test output shows `"Generated new Session ID: wake_the...9189"` — the `redact_uuid`
function shows `"wake_the"` as the first 8 chars (from `"wake_the_claude:"` debug prefix).

**Affected tests** (all 10 failures):
- `test_session_id_without_value_uses_uuid_fallback_when_uuidgen_fails`
- `test_session_id_without_value_uses_python_fallback_when_uuidgen_and_proc_fail`
- `test_session_id_autogenerated_flag_and_value_are_separate`
- `test_default_interactive_script_forwards_expected_defaults`
- `test_default_launcher_does_not_skip_permissions`
- `test_default_launcher_executes_with_expected_default_arguments`
- `test_default_launcher_forwards_skip_permissions_when_opted_in`
- `test_default_launcher_opt_in_skip_permissions`
- `test_default_launcher_runtime_honors_skip_permissions_opt_in`
- `test_default_launcher_runtime_omits_skip_permissions_by_default`

The last 7 all use the default interactive launcher which passes `--id` without a value,
triggering the `generate_uuid()` code path.

**Fix**:
1. Add `echo "${generated_uuid}"` before each `return "${TRUE}"` in `generate_uuid()`
2. Redirect all `debug_log` calls inside `generate_uuid()` to stderr (`>&2`) so they don't
   contaminate the captured stdout

### Bug 2: Default launcher unconditionally exits 0 (SECONDARY)

**File**: `scripts/default_interactive_session_claude_code.bash`, line 47

**Problem**: The script ends with `exit 0` regardless of whether `wake_the_claude.bash` succeeded
or failed. This masks inner failures.

**Impact**: Currently doesn't cause test failures directly (tests check returncode == 0 which
passes due to this bug). Once Bug 1 is fixed, the inner script will succeed so `exit 0` won't
mask anything. However, this remains a correctness defect — if `wake_the_claude.bash` fails in
production, the caller would see success.

**Fix**: Capture the exit code from `wake_the_claude.bash` and propagate it.

## Implementation Plan

### Step 1: Fix `generate_uuid()` in `wake_the_claude.bash`

For each of the three fallback paths (uuidgen, /proc, python3), change:
```bash
debug_log "Generated UUID: ${generated_uuid}"
return "${TRUE}"
```
To:
```bash
debug_log "Generated UUID: ${generated_uuid}" >&2
echo "${generated_uuid}"
return "${TRUE}"
```

### Step 2: Fix exit code propagation in `default_interactive_session_claude_code.bash`

Change:
```bash
"${SCRIPT_PATH}/wake_the_claude.bash" "${CLAUDE_ARGS[@]}"
echo "Closed Default Interactive session with Claude Code"
exit 0
```
To:
```bash
"${SCRIPT_PATH}/wake_the_claude.bash" "${CLAUDE_ARGS[@]}"
EXIT_CODE=$?
echo "Closed Default Interactive session with Claude Code"
exit "${EXIT_CODE}"
```

### Step 3: Run full test suite and verify all 10 failures are resolved

```bash
python3 -m pytest tests/test_wake_the_claude.py -v
```

Expected result: 0 failures, 58 tests passing.
